1. DESCRIZIONE GENERALE
-----------------------
"Gattile San Paolo" è un sito web per la gestione di adozioni e volontariato
in un gattile di Torino. Permette agli utenti di sfogliare i gatti ospiti,
prenotare visite conoscitive e registrarsi come volontari. Gli amministratori
possono inserire nuovi animali nella struttura.

Il sito è sviluppato in PHP 8+, con componenti React per la visualizzazione
dei gatti e Vanilla JavaScript per la gestione dei form.

3. COMUNICAZIONE REACT ↔ VANILLA JS (CustomEvent)
-------------------------------------------------
La sezione di visualizzazione gatti (gatti.php) usa due tecnologie distinte:
- Il componente React gestisce la lista, il filtraggio e la selezione.
- Il form di prenotazione visita è in Vanilla JavaScript puro.

Per comunicare la lista di gatti selezionati dal componente React al form
Vanilla JS, si è scelto un CustomEvent DOM:

    document.dispatchEvent(new CustomEvent('gattiSelezionatiAggiornati', {
        detail: { gatti: [...] },
        bubbles: true,
    }));

MOTIVAZIONE: il CustomEvent è uno standard Web nativo (W3C), supportato da
tutti i browser moderni. Permette comunicazione unidirezionale (React → VanillaJS)
senza:
- condivisione di variabili globali (evita inquinamento del namespace globale)
- dipendenza da framework comuni o librerie esterne
- accoppiamento diretto tra i due moduli (ciascuno è autonomamente testabile)

Alternative considerate e scartate:
- Variabile globale window.gattiSelezionati: rischio di collisioni, difficile
  da tracciare, non standard.
- Callback diretta: richiederebbe che React conosca il form VanillaJS,
  violando il principio di separazione delle responsabilità.
- Redux/Context: sovra-ingegnerizzazione per una comunicazione così semplice
  e unidirezionale.

Il CustomEvent consente anche a eventuali futuri moduli di iscriversi allo
stesso evento senza modificare il componente React (open/closed principle).


4. SEPARAZIONE DEI PRIVILEGI DB
-------------------------------
Si usano tre utenti MySQL distinti, con il principio del minimo privilegio
(OWASP A01:2021 — Broken Access Control):

a) 'lecture'@'localhost' — Solo SELECT su tutto gattile_db
   Usato da: index.php (nuovi arrivi), api/gatti.php, parti di auth.php,
             turni.php (conteggio), login (verifica username prima di insert).
   Rationale: le pagine di sola lettura non devono mai poter modificare dati.

b) 'modifier'@'localhost' — SELECT, INSERT, UPDATE su tutto gattile_db
   Usato da: inserisci_gatto.php, api/prenota_visita.php, api/turni.php (POST).
   Rationale: le operazioni di business logic richiedono lettura e scrittura,
              ma mai eliminazione di record (DROP/DELETE non concesso).

c) 'registrator'@'localhost' — Solo INSERT su gattile_db
   Usato da: registrazione.php (solo per inserire il nuovo utente).
   Rationale: la pagina di registrazione non deve poter leggere altri dati
              utente né modificare record esistenti. È il livello di privilegio
              minimo indispensabile per la sua funzione.

In nessun caso viene usato l'utente root/admin di sistema, né un utente
con DROP, DELETE o GRANT.

La distinzione tra lettore/modificatore/registratore è implementata nella
funzione getDB($role) in includes/db.php, che accetta un parametro stringa
e seleziona le credenziali appropriate. Ogni endpoint API e ogni pagina PHP
chiama getDB() con il ruolo minimo necessario.


5. AUTENTICAZIONE E "RICORDAMI"
--------------------------------
L'autenticazione avviene sempre via sessione PHP con verifica della password
(password_verify + Bcrypt). Il cookie "ricordami" (gattile_remember) contiene
un token opaco a 64 caratteri esadecimali (32 byte casuali via random_bytes).

Il token è associato lato server (in sessione) allo username corrispondente.
In una versione produzione il mapping andrebbe in una tabella DB dedicata
(remember_tokens) con colonne: token_hash, utente_id, scadenza.

Il cookie NON contiene mai le credenziali in chiaro. La sua unica funzione
è precompilare il campo username nel form di login; l'accesso rimane
subordinato alla verifica della password.

Attributi del cookie:
- HttpOnly: non accessibile a JavaScript (prevenzione XSS)
- SameSite=Strict: prevenzione CSRF
- Secure: attivo in HTTPS (condizionale su $_SERVER['HTTPS'])
- Durata: 72 ore


6. VALIDAZIONE E SICUREZZA (OWASP)
------------------------------------
A. Input: tutti i dati POST sono filtrati con filter_input() e poi validati
   con logica specifica (lunghezze, pattern, range numerici).

B. Output: ogni dato utente viene passato per esc() (htmlspecialchars con
   ENT_QUOTES | ENT_SUBSTITUTE) prima dell'output HTML.

C. SQL Injection: mai interpolazione diretta. Tutte le query usano prepared
   statements PDO con parametri bindati.

D. CSRF: il sito usa SameSite=Strict sui cookie di sessione e referrer-check
   (implementazione base; in produzione aggiungere token CSRF esplicito nel form).

E. Password: hashate con password_hash(PASSWORD_BCRYPT) e mai conservate
   in chiaro né in sessione.

F. Session fixation: session_regenerate_id(true) dopo ogni login riuscito.

G. Content-Type sniffing: header X-Content-Type-Options: nosniff sugli endpoint JSON.

H. Informazioni di errore: i messaggi mostrati all'utente sono generici;
   i dettagli tecnici sono scritti solo in error_log (lato server).


7. ACCESSIBILITÀ (WCAG 2.1 AA)
--------------------------------
- Tutti i colori rispettano rapporto di contrasto ≥ 4.5:1 (testo normale)
  o ≥ 3:1 (testo grande). Verificato con la palette:
  - testo #2d1f12 su sfondo #faf6f0: rapporto ~11:1 ✓
  - primario #7b4b2a su sfondo #faf6f0: rapporto ~5.8:1 ✓
  - accento #d4813a su sfondo #faf6f0: rapporto ~3.2:1 (usato solo per testo grande) ✓

- Skip link "Vai al contenuto principale" visibile al focus.

- Tutti i form hanno <label for=""> associato all'input.

- Errori inline con role="alert" e aria-live="polite/assertive".

- Immagini con attributo alt descrittivo (placeholder con descrizione
  contestualizzata al nome del gatto).

- Elementi interattivi React con role, aria-checked, tabIndex, gestione keydown.

- <main tabindex="-1"> per ricevere il focus dal skip link.

- Tag semantici: <header>, <nav>, <main>, <section>, <article>, <aside>,
  <footer>, <figure>, <figcaption>, <time>, <data>, <meter>, <progress>,
  <details>, <summary>, <abbr>, <strong>, <em>, <cite>, <code>.

- Nessun uso di <div> o <span> dove esiste un elemento semantico appropriato.

- Un solo <h1> per pagina, gerarchia rispettata.

- prefers-reduced-motion rispettato: le animazioni CSS vengono annullate.


8. CSS E SEPARAZIONE DEI LINGUAGGI
------------------------------------
- Tutto il presentazionale è nel CSS (stile.css e stampa.css).
- Nessun attributo HTML di stile inline (style="..." non viene mai usato
  direttamente nelle pagine PHP; solo raramente nello JS per aggiornamenti
  dinamici strettamente necessari).
- Nessun tag obsoleto (<font>, <center>, <big>, <tt>).
- Layout mobile-first: le media query espandono verso schermi più grandi
  (min-width: 640px, 900px).
- Variabili CSS per colori, spaziature e tipografia: coerenza garantita.

CSS di stampa (stampa.css, media="print"):
- Nasconde navigazione, banner cookie, pulsanti, form.
- Mostra URL dei link tra parentesi via ::after content: attr(href).
- Sottolinea tutti i link.
- Tipografia con font serif (Georgia) per leggibilità su carta.
- Stile tabelle con bordi visibili.
- Apertura forzata dei <details>.


9. PERFORMANCE E ROBUSTEZZA
-----------------------------
- Le immagini dei gatti sono lazy-loaded (loading="lazy").
- Le immagini sono SVG scalabili, leggere e senza dipendenze esterne.
- Il componente React è caricato via CDN con crossorigin per integrità
  (in produzione usare hash integrity + bundler).
- PHP usa declare(strict_types=1) in ogni file per type safety.
- Tutti i blocchi DB sono in try/catch con fallback all'utente.
- Il form di prenotazione è inaccessibile via JS se non si è loggati
  (controllo PHP + controllo API).
- Le API JSON restituiscono codici HTTP appropriati (200, 400, 401, 403, 405, 409, 500).
